Mental Model
Understand Kafka before you configure it
TRADITIONAL REST Service A ──HTTP──> Service B ──> Response ──> Service A (Service A waits. If B is down — A fails.) KAFKA EVENT STREAMING Service A ──> [Topic: order-events] ──> Service B [Topic: order-events] ──> Service C [Topic: order-events] ──> Service D (A publishes and moves on. B,C,D read independently at their own pace.)
💡 Core idea: Kafka is a distributed, ordered, immutable log. Producers append to it. Consumers read from it at their own pace. Messages are never deleted on read — they expire based on retention policy.

Producer Properties
All properties under spring.kafka.producer
acks all CRITICAL
▶
How many broker replicas must confirm receipt before producer considers send successful. Controls durability vs throughput tradeoff.
ValueMeaningUse when
0Fire and forget. No confirmation. Fastest.Metrics, logs — loss acceptable
1Leader confirms only. Fast, small risk.Moderate importance events
all or -1All in-sync replicas confirm. Safest. BANKINGPayments, orders, critical data
retries 3 IMPORTANT
▶
How many times producer retries on retriable failures (network timeout, leader election). Does NOT retry on non-retriable errors (message too large, authorization).
ValueMeaning
0No retry — fail immediately
3Retry 3 times — good balance RECOMMENDED
Integer.MAX_VALUERetry forever — dangerous, can cause ordering issues without idempotence
properties.enable.idempotence true CRITICAL
▶
Prevents duplicate messages when producer retries. Kafka assigns each producer a unique ID and sequence number — detects and discards duplicates automatically.
✅ idempotence=true
Producer sends → network fails
Producer retries same message
Kafka detects duplicate (same seq no.)
Kafka discards duplicate
Result: exactly ONE message stored
❌ idempotence=false
Producer sends → network fails
Producer retries same message
Kafka has no way to detect duplicate
Result: TWO identical messages stored
Customer charged twice!
🚨 Requires: acks=all, retries > 0, max.in.flight.requests.per.connection ≤ 5. Always enable for financial systems.

Consumer Properties
All properties under spring.kafka.consumer
group-id CRITICAL
▶
Name of the consumer group. Kafka tracks read offset per group independently. Multiple instances of same service use same group-id — Kafka distributes partitions between them.
PatternExampleEffect
Service name RECOMMENDEDfraud-detection-serviceAll instances of this service share the work
Use application name${spring.application.name}Auto uses service name from config
Different group = independent readfraud-service vs notification-serviceBoth read same topic from beginning independently
💡 Two services with different group-ids both consuming order-events will EACH get EVERY message. Kafka fans out to groups, not individual consumers.
auto-offset-reset latest CRITICAL
▶
Where consumer starts reading when no committed offset exists yet — first time service starts or when offset has expired.
ValueBehaviourUse when
earliestRead from very beginning of topicYou want to process ALL existing messages. Critical services. BANKING
latestRead only new messages from nowReal-time only — don't care about past
noneThrow exception if no offset foundYou must have committed offsets — explicit error on missing
enable-auto-commit false CRITICAL — ALWAYS FALSE
▶
Should Kafka automatically commit offset after receiving message — regardless of whether processing succeeded.
❌ enable-auto-commit: true
Message received
Offset auto-committed (5 seconds)
Processing starts
Service crashes mid-processing
Message marked as "done"
Data silently lost — never reprocessed
✅ enable-auto-commit: false
Message received
Processing starts
Processing succeeds
ack.acknowledge() called
Offset committed
Service crashes? Message redelivered.
🚨 Production rule: ALWAYS set enable-auto-commit: false. Use manual acknowledgement. This is the #1 source of data loss in Kafka beginner projects.
properties.max.poll.records 500 IMPORTANT
▶
Maximum number of records returned in a single poll call. Controls how many messages consumer processes per batch.
ValueUse when
10Slow processing (AI calls, external API) — process fewer, avoid timeout
100Moderate processing — good balance
500Fast processing (simple DB writes) — default
properties.max.poll.interval.ms 300000 IMPORTANT
▶
Maximum time between poll() calls before Kafka considers consumer dead and triggers rebalance. If your processing takes longer than this — increase it.
ScenarioValue
Fast processing30000 (30s)
Default300000 (5 min) DEFAULT
AI calls / slow APIs600000 (10 min)
⚠️ If your AI fraud analysis takes 30 seconds per message and you poll 10 messages — total processing = 300s. If max.poll.interval.ms is less — Kafka thinks consumer died and triggers rebalance.
properties.spring.json.trusted.packages CRITICAL — SECURITY
▶
Which Java packages JsonDeserializer is allowed to deserialize into objects. Prevents malicious messages from instantiating arbitrary classes.
❌ NEVER do this
spring.json.trusted.packages: "*"
# Allows ANY class to be deserialized
# Security vulnerability
✅ Always be specific
spring.json.trusted.packages: "com.neobank.events"
# Only your event classes
# Safe and explicit

Topic Configuration
How to create and configure topics in Spring Boot
KafkaTopicConfig.java — production pattern
@Configuration
public class KafkaTopicConfig {

  @Bean
  public KafkaAdmin.NewTopics allTopics() {
    return new KafkaAdmin.NewTopics(

      // Standard event topic
      TopicBuilder.name("order-events")
        .partitions(6)      // 2x expected max consumers
        .replicas(1)       // 3 in production
        .config(TopicConfig.RETENTION_MS_CONFIG, "604800000")   // 7 days
        .config(TopicConfig.CLEANUP_POLICY_CONFIG, "delete")
        .build(),

      // High priority topic — more partitions
      TopicBuilder.name("payment-events")
        .partitions(12)
        .replicas(1)
        .config(TopicConfig.RETENTION_MS_CONFIG, "2592000000")  // 30 days
        .build(),

      // Compact topic — keep latest value per key (e.g. user profiles)
      TopicBuilder.name("user-profile")
        .partitions(3)
        .replicas(1)
        .config(TopicConfig.CLEANUP_POLICY_CONFIG, "compact")
        .config(TopicConfig.MIN_CLEANABLE_DIRTY_RATIO_CONFIG, "0.1")
        .build(),

      // Dead Letter Topic — longer retention
      TopicBuilder.name("order-events.DLT")
        .partitions(1)
        .replicas(1)
        .config(TopicConfig.RETENTION_MS_CONFIG, "2592000000")  // 30 days
        .build()
    );
  }
}

Producer Code Patterns
Every producer pattern you will use in real projects
Pattern 1 — Basic async producer (most common)
@Service
@Slf4j
@RequiredArgsConstructor
public class OrderEventProducer {

  private final KafkaTemplate<String, OrderEvent> kafkaTemplate;

  public void publish(OrderEvent event) {
    kafkaTemplate
      .send("order-events", event.getOrderId(), event)
      .whenComplete((result, ex) -> {
        if (ex != null) {
          log.error("Failed to publish orderId={}", event.getOrderId(), ex);
        } else {
          log.info("Published orderId={} partition={} offset={}",
            event.getOrderId(),
            result.getRecordMetadata().partition(),
            result.getRecordMetadata().offset());
        }
      });
  }
}
Pattern 2 — Sync producer (wait for confirmation)
public RecordMetadata publishSync(OrderEvent event) throws Exception {
  return kafkaTemplate
    .send("order-events", event.getOrderId(), event)
    .get(10, TimeUnit.SECONDS)  // block max 10s
    .getRecordMetadata();
  // Use only when caller MUST know message was stored
}
Pattern 3 — Producer with headers (correlationId tracing)
public void publishWithHeaders(OrderEvent event, String correlationId) {
  ProducerRecord<String, OrderEvent> record =
    new ProducerRecord<>("order-events", event.getOrderId(), event);

  record.headers()
    .add("correlationId", correlationId.getBytes())
    .add("sourceService", "order-service".getBytes())
    .add("timestamp", Instant.now().toString().getBytes());

  kafkaTemplate.send(record)
    .whenComplete((result, ex) -> {
      if (ex != null) log.error("Send failed", ex);
    });
}
Pattern 4 — Generic producer (one KafkaTemplate for all event types)
@Service
@Slf4j
@RequiredArgsConstructor
public class EventProducer {

  // KafkaTemplate<String, Object> handles any event type
  private final KafkaTemplate<String, Object> kafkaTemplate;

  public void send(String topic, String key, Object event) {
    kafkaTemplate.send(topic, key, event)
      .whenComplete((result, ex) -> {
        if (ex != null)
          log.error("Failed topic={} key={}", topic, key, ex);
        else
          log.info("Sent topic={} partition={} offset={}",
            topic,
            result.getRecordMetadata().partition(),
            result.getRecordMetadata().offset());
      });
  }
}

Consumer Code Patterns
Every consumer pattern you will use in real projects
Pattern 1 — Manual ack consumer (production standard)
@KafkaListener(topics = "order-events", groupId = "fraud-service")
public void handle(
    @Payload OrderEvent event,
    @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
    @Header(KafkaHeaders.OFFSET) long offset,
    Acknowledgment ack) {

  log.info("Received orderId={} partition={} offset={}",
    event.getOrderId(), partition, offset);

  try {
    processOrder(event);     // your business logic
    ack.acknowledge();       // ✅ commit ONLY after success
  } catch (Exception e) {
    log.error("Failed orderId={}", event.getOrderId(), e);
    // ❌ do NOT ack — Kafka redelivers
  }
}
Pattern 2 — Batch consumer (high throughput)
@KafkaListener(topics = "order-events", groupId = "analytics-service")
public void handleBatch(
    List<OrderEvent> events,
    Acknowledgment ack) {

  log.info("Processing batch of {} events", events.size());
  events.forEach(this::processOrder);
  ack.acknowledge();
}

Error Handling & Dead Letter Topic
Production-grade error handling — never lose a message
MESSAGE LIFECYCLE WITH ERROR HANDLING Message arrives at consumer │ ▼ Processing attempt 1 → FAILS │ ▼ (wait 2 seconds) Processing attempt 2 → FAILS │ ▼ (wait 4 seconds) Processing attempt 3 → FAILS │ ▼ (max attempts reached) DeadLetterPublishingRecoverer │ ▼ Message sent to order-events.DLT │ ▼ Original consumer moves on to next message DLT consumer handles failed message separately
KafkaErrorHandlerConfig.java — complete production setup
@Configuration
@Slf4j
public class KafkaErrorHandlerConfig {

  @Bean
  public DefaultErrorHandler errorHandler(
      KafkaTemplate<String, Object> kafkaTemplate) {

    // Routes failed messages to topic-name.DLT automatically
    var recoverer = new DeadLetterPublishingRecoverer(kafkaTemplate);

    // Retry 3 times: 2s → 4s → 8s then give up
    var backoff = new ExponentialBackOff(2_000L, 2.0);
    backoff.setMaxAttempts(3);

    var handler = new DefaultErrorHandler(recoverer, backoff);

    // Log each retry attempt
    handler.setRetryListeners((record, ex, attempt) ->
      log.warn("Retry attempt={} topic={} offset={} error={}",
        attempt, record.topic(), record.offset(), ex.getMessage()));

    // Do NOT retry these exceptions — go straight to DLT
    handler.addNotRetryableExceptions(
      IllegalArgumentException.class,   // bad data — retrying won't help
      NullPointerException.class         // code bug — retrying won't help
    );

    return handler;
  }
}

Kafka Transactions
Exactly-once semantics across consume → process → produce
💡 When to use Kafka transactions: When you need to guarantee that consuming a message AND producing a result happen atomically. If your service crashes between consume and produce — the consumed message is rolled back and redelivered.
Kafka Transactions — application.yml
spring:
  kafka:
    producer:
      transaction-id-prefix: fraud-service-tx-  # enables transactions
      properties:
        enable.idempotence: true

# Kafka Streams transactions
    streams:
      properties:
        processing.guarantee: exactly_once_v2
Transactional consume → produce pattern
@KafkaListener(topics = "order-events")
@Transactional("kafkaTransactionManager")
public void processAndForward(OrderEvent event, Acknowledgment ack) {
  // If anything below fails — BOTH consume and produce are rolled back
  FraudResult result = fraudService.analyze(event);
  kafkaTemplate.send("fraud-results", event.getOrderId(), result);
  ack.acknowledge();
  // Atomic: either both happen or neither does
}

Delivery Guarantees
At-most-once vs At-least-once vs Exactly-once
GuaranteeWhat it meansConfigUse when
At most once Message delivered 0 or 1 times. May be lost. acks=0, auto-commit=true Metrics, logs — loss acceptable
At least once Message delivered 1+ times. No loss, possible duplicates. acks=all, manual ack, retries Most business events — handle idempotency in consumer
Exactly once Message delivered exactly 1 time. No loss, no duplicates. idempotence=true + transactions Financial transactions, payments

Partitioning Strategy
How to choose the right message key
Key choiceEffectUse when
orderIdAll events for same order go to same partition — orderedOrder events — status must be in order
accountIdAll transactions for same account orderedBanking — debit/credit sequence matters
userIdAll user events orderedUser activity tracking
nullRound-robin across partitions — no orderingEvents with no ordering requirement
random UUIDEvenly distributed — maximum parallelismHigh throughput, order doesn't matter
💡 Rule: Same key = same partition = ordered delivery. If two events must be processed in order — they must have the same key.

Pattern 1 — Filter and transform (stateless)
@Configuration
@EnableKafkaStreams
public class TransactionStreamTopology {

  @Bean
  public KStream<String, TransactionEvent> transactionStream(
      StreamsBuilder builder) {

    KStream<String, TransactionEvent> stream =
      builder.stream("transaction-completed");

    // Filter — only DEBIT transactions
    KStream<String, TransactionEvent> debits = stream
      .filter((key, txn) -> "DEBIT".equals(txn.getType()));

    // Transform — add category label
    KStream<String, EnrichedTransaction> enriched = debits
      .mapValues(txn -> EnrichedTransaction.builder()
        .transactionId(txn.getTransactionId())
        .amount(txn.getAmount())
        .category(categorize(txn.getMerchant()))
        .build());

    // Route to different topics based on amount
    enriched.split()
      .branch((key, txn) -> txn.getAmount().compareTo(new BigDecimal("10000")) > 0,
        Branched.withConsumer(s -> s.to("high-value-transactions")))
      .defaultBranch(
        Branched.withConsumer(s -> s.to("normal-transactions")));

    return stream;
  }
}
Pattern 2 — Aggregation with windowing (stateful)
@Bean
public KTable<Windowed<String>, Long> spendingByAccountPerHour(
    StreamsBuilder builder) {

  return builder
    .stream("transaction-completed",
      Consumed.with(Serdes.String(), new JsonSerde<>(TransactionEvent.class)))

    // Keep only debits
    .filter((key, txn) -> "DEBIT".equals(txn.getType()))

    // Re-key by accountId (for grouping)
    .selectKey((key, txn) -> txn.getAccountId())

    // Group by accountId
    .groupByKey()

    // Count within 1-hour tumbling window
    .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofHours(1)))

    // Count transactions per account per hour
    .count(Materialized.as("txn-count-per-hour"));
    // Results written to changelog topic automatically
}
Pattern 3 — Stream-Table Join (enrich stream with DB data)
@Bean
public KStream<String, EnrichedTransaction> enrichedStream(
    StreamsBuilder builder) {

  // Stream of transactions
  KStream<String, TransactionEvent> transactions =
    builder.stream("transaction-completed");

  // Table of account info (compacted topic — latest state per accountId)
  KTable<String, AccountInfo> accounts =
    builder.table("account-info",
      Materialized.as("account-store"));

  // Join: enrich each transaction with account details
  return transactions.join(
    accounts,
    (txn, account) -> EnrichedTransaction.builder()
      .transactionId(txn.getTransactionId())
      .amount(txn.getAmount())
      .customerName(account.getCustomerName())
      .accountType(account.getAccountType())
      .build()
  );
}
Pattern 4 — Stream-Stream Join (correlate two event streams)
@Bean
public KStream<String, PaymentConfirmation> paymentConfirmations(
    StreamsBuilder builder) {

  KStream<String, OrderEvent> orders =
    builder.stream("order-events");

  KStream<String, PaymentEvent> payments =
    builder.stream("payment-events");

  // Join orders with payments within 5-minute window
  // Both must have same key (orderId)
  return orders.join(
    payments,
    (order, payment) -> PaymentConfirmation.builder()
      .orderId(order.getOrderId())
      .amount(payment.getAmount())
      .status("CONFIRMED")
      .build(),
    JoinWindows.ofTimeDifferenceWithNoGrace(Duration.ofMinutes(5))
  );
}

Schema Registry + Avro
Enterprise event contracts — type-safe, versioned, backward compatible
💡 Why Schema Registry: JSON has no schema enforcement. If producer changes event structure — consumer crashes silently. Schema Registry enforces contracts between services and enables safe schema evolution.
WITHOUT SCHEMA REGISTRY Producer changes OrderEvent → removes "amount" field Consumer tries to read "amount" → NullPointerException at 2AM in production WITH SCHEMA REGISTRY Producer tries to remove "amount" field Schema Registry rejects: "BREAKING CHANGE — consumers depend on this field" Producer must use backward-compatible change or bump schema version

Transactional Outbox Pattern
Solve the dual-write problem — DB + Kafka atomically
🚨 The dual-write problem: You save order to DB then publish to Kafka. If service crashes between these two — DB has order but Kafka has no event. Downstream services never know the order was created. Outbox pattern solves this.
WITHOUT OUTBOX (dangerous) Save order to DB ← step 1 Publish to Kafka ← step 2 — service crashes HERE Notification service never gets the event Customer never gets confirmation WITH OUTBOX (safe) BEGIN TRANSACTION Save order to DB Save event to outbox_events table ← same transaction COMMIT ← atomic — both or neither Outbox poller reads outbox_events table Publishes to Kafka Marks outbox event as published
Outbox table — SQL
CREATE TABLE outbox_events (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  aggregate_id  VARCHAR(255) NOT NULL,    -- orderId, accountId etc
  event_type    VARCHAR(100) NOT NULL,    -- "ORDER_CREATED"
  topic         VARCHAR(200) NOT NULL,    -- Kafka topic to publish to
  payload       JSONB         NOT NULL,   -- the event JSON
  status        VARCHAR(20)  DEFAULT 'PENDING', -- PENDING | PUBLISHED | FAILED
  created_at    TIMESTAMP    DEFAULT NOW(),
  published_at  TIMESTAMP
);
OrderService.java — save order + outbox in one transaction
@Service
@RequiredArgsConstructor
public class OrderService {

  private final OrderRepository orderRepo;
  private final OutboxRepository outboxRepo;
  private final ObjectMapper objectMapper;

  @Transactional  // ← SINGLE transaction covers BOTH DB writes
  public Order createOrder(CreateOrderRequest req) throws Exception {

    // Step 1 — save business data
    Order order = orderRepo.save(Order.from(req));

    // Step 2 — save event in SAME transaction
    OrderCreatedEvent event = OrderCreatedEvent.builder()
      .orderId(order.getId())
      .customerId(order.getCustomerId())
      .amount(order.getAmount())
      .build();

    OutboxEvent outbox = OutboxEvent.builder()
      .aggregateId(order.getId())
      .eventType("ORDER_CREATED")
      .topic("order-events")
      .payload(objectMapper.writeValueAsString(event))
      .status("PENDING")
      .build();

    outboxRepo.save(outbox);

    // If ANYTHING above fails — BOTH are rolled back
    // No orphaned DB records with missing Kafka events
    return order;
  }
}
OutboxPoller.java — publishes pending events to Kafka
@Component
@RequiredArgsConstructor
@Slf4j
public class OutboxPoller {

  private final OutboxRepository outboxRepo;
  private final KafkaTemplate<String, String> kafkaTemplate;

  @Scheduled(fixedDelay = 1000)  // poll every 1 second
  @Transactional
  public void publishPendingEvents() {
    List<OutboxEvent> pending = outboxRepo
      .findTop100ByStatusOrderByCreatedAtAsc("PENDING");

    pending.forEach(event -> {
      try {
        kafkaTemplate.send(event.getTopic(), event.getAggregateId(),
          event.getPayload()).get();  // sync — wait for confirmation

        event.setStatus("PUBLISHED");
        event.setPublishedAt(Instant.now());
        outboxRepo.save(event);

      } catch (Exception e) {
        log.error("Failed to publish outbox event id={}", event.getId(), e);
        event.setStatus("FAILED");
        outboxRepo.save(event);
      }
    });
  }
}

Saga Pattern — Distributed Transactions
Coordinate multi-service transactions without 2-phase commit
💡 When to use Saga: When a business operation spans multiple services and all must succeed or all must compensate (rollback). Example: place order → reserve inventory → charge payment → all three must succeed or all three must undo.
SAGA CHOREOGRAPHY (Kafka-based — each service reacts to events) ORDER PLACED │ ▼ [order-events] → Inventory Service → reserve stock │ RESERVED → [inventory-events] → Payment Service → charge card FAILED → [inventory-events] → Order Service → cancel order │ CHARGED → [payment-events] → Order Service → confirm FAILED → [payment-events] → Inventory Service → release stock → Order Service → cancel order
OrderSagaOrchestrator.java — choreography via Kafka
@Service
@Slf4j
@RequiredArgsConstructor
public class OrderSagaOrchestrator {

  private final KafkaTemplate<String, Object> kafkaTemplate;
  private final OrderRepository orderRepo;

  // Step 1 — Start saga: order placed → trigger inventory reservation
  public void startOrderSaga(Order order) {
    kafkaTemplate.send("inventory-commands", order.getId(),
      ReserveInventoryCommand.builder()
        .orderId(order.getId())
        .productId(order.getProductId())
        .quantity(order.getQuantity())
        .build());
  }

  // Step 2a — Inventory reserved → trigger payment
  @KafkaListener(topics = "inventory-events", groupId = "order-saga")
  public void onInventoryEvent(InventoryEvent event, Acknowledgment ack) {
    if ("RESERVED".equals(event.getStatus())) {
      kafkaTemplate.send("payment-commands", event.getOrderId(),
        ChargePaymentCommand.builder()
          .orderId(event.getOrderId())
          .amount(event.getAmount())
          .build());
    } else if ("FAILED".equals(event.getStatus())) {
      // Compensate — cancel the order
      orderRepo.updateStatus(event.getOrderId(), "CANCELLED");
      log.warn("Saga failed at inventory step for orderId={}",
        event.getOrderId());
    }
    ack.acknowledge();
  }

  // Step 2b — Payment result → complete or compensate
  @KafkaListener(topics = "payment-events", groupId = "order-saga")
  public void onPaymentEvent(PaymentEvent event, Acknowledgment ack) {
    if ("CHARGED".equals(event.getStatus())) {
      orderRepo.updateStatus(event.getOrderId(), "CONFIRMED");
    } else {
      // Compensate — release inventory, cancel order
      kafkaTemplate.send("inventory-commands", event.getOrderId(),
        ReleaseInventoryCommand.of(event.getOrderId()));
      orderRepo.updateStatus(event.getOrderId(), "CANCELLED");
    }
    ack.acknowledge();
  }
}

CQRS + Event Sourcing with Kafka
Separate reads from writes — Kafka as the event log
CQRS PATTERN COMMAND SIDE (writes) POST /orders → OrderService → saves to write DB → publishes OrderCreatedEvent QUERY SIDE (reads) [order-events] → ProjectionBuilder (Kafka Consumer) → builds read-optimised view in read DB GET /orders/{id} → reads from read DB directly — no joins, no latency

@Service
@RequiredArgsConstructor
public class OrderCommandHandler {

  private final OrderWriteRepository writeRepo;
  private final KafkaTemplate<String, Object> kafkaTemplate;

  @Transactional
  public String handle(CreateOrderCommand cmd) {
    // 1. Apply business rules
    Order order = Order.create(cmd);

    // 2. Save to write DB
    writeRepo.save(order);

    // 3. Publish event — read side will project it

    return order.getId();
  }
}
OrderProjection.java — read side projection builder
@Component
@RequiredArgsConstructor
public class OrderProjection {

  private final OrderReadRepository readRepo;

  // Listens to ALL order events and builds read model
  @KafkaListener(topics = {
    "order-events",
    "payment-events",
    "inventory-events"
  }, groupId = "order-projection")
  public void project(
      @Payload String rawEvent,
      @Header(KafkaHeaders.RECEIVED_TOPIC) String topic,
      Acknowledgment ack) {

    switch (topic) {
      case "order-events" -> handleOrderEvent(rawEvent);
      case "payment-events" -> handlePaymentEvent(rawEvent);
      case "inventory-events" -> handleInventoryEvent(rawEvent);
    }
    ack.acknowledge();
  }

  private void handleOrderEvent(String rawEvent) {
    // Parse and upsert into read-optimised view
    OrderReadModel model = OrderReadModel.from(rawEvent);
    readRepo.upsert(model);
  }
}

Top 30 Kafka Interview Questions
Senior developer and architect level — with answers
Q1. What is the difference between a Kafka partition and a topic?
▶
Topic is a logical stream name — like a database table name. Partition is the physical storage unit — the actual ordered, immutable log. A topic with 3 partitions is 3 separate ordered logs. Kafka writes to partitions, not topics. Same key always goes to same partition — ordering is guaranteed per partition, not per topic.
Q2. You have 3 partitions and 5 consumers in the same group. What happens?
▶
Only 3 consumers are active — one per partition. The other 2 sit idle waiting. Partitions are the unit of parallelism. You can never have more active consumers than partitions in a group. To scale consumption — increase partitions first, then add consumers.
Q3. What is enable-auto-commit and why should it always be false in production?
▶
Auto-commit commits the offset automatically every 5 seconds — regardless of whether your business logic succeeded. If service crashes between commit and processing — message is marked done but was never processed. Data silently lost. Always use enable-auto-commit: false with manual ack.acknowledge() called only after successful processing.
Q4. What is a consumer group rebalance and when does it happen?
▶
Rebalance is Kafka redistributing partition ownership among consumers. Happens when: consumer joins the group, consumer leaves or crashes, new partitions are added. During rebalance — all consumers stop consuming temporarily. Minimize rebalances by: setting appropriate session.timeout.ms, increasing max.poll.interval.ms if processing is slow, using static group membership.
Q5. How do you guarantee message ordering in Kafka?
▶
Three requirements: (1) Same key → same partition (key routes to partition via hash). (2) Single consumer per partition in a consumer group. (3) max.in.flight.requests.per.connection=1 OR enable.idempotence=true with up to 5 in-flight. Ordering is guaranteed per partition only — not across partitions.
Q6. What is the difference between at-most-once, at-least-once, exactly-once?
▶
At-most-once: acks=0, auto-commit. Message may be lost. Never use for business data. At-least-once: acks=all, manual ack, retries. Message never lost but may be duplicated on retry — consumer must be idempotent. Exactly-once: idempotent producer + Kafka transactions. No loss, no duplicates. Most expensive. Use for financial data.
Q7. What is a Dead Letter Topic and how do you implement it in Spring Boot?
▶
DLT is a topic where permanently failed messages are parked after all retries exhausted. Prevents one bad message from blocking the entire partition. In Spring Boot: configure DefaultErrorHandler with DeadLetterPublishingRecoverer and ExponentialBackOff. Failed messages automatically go to originalTopic.DLT. Implement a separate @KafkaListener for the DLT topic to alert teams and investigate.
Q8. What is the Transactional Outbox Pattern and when do you need it?
▶
Use when you need to save to DB and publish to Kafka atomically. Problem: if you save to DB then crash before Kafka publish — data inconsistency. Solution: save both the business data AND the event to an outbox_events table in one @Transactional. A separate poller reads the outbox table and publishes to Kafka. This is the recommended pattern for all event-driven banking systems.
Q9. What is consumer lag and how do you monitor it?
▶
Consumer lag = difference between latest message offset and consumer's committed offset. Lag=0 means consumer is caught up. Growing lag means consumer is falling behind — either too slow or too few instances. Monitor via: kafka-consumer-groups.sh --describe, Kafka UI, or kafka_consumer_group_lag metric in Prometheus. Alert when lag grows continuously.
Q10. How does Kafka KRaft mode differ from Zookeeper mode?
▶
Zookeeper mode: Kafka required an external Zookeeper cluster for metadata management, leader election, and configuration. Two systems to operate, two failure modes. KRaft mode (since Kafka 3.3): Kafka manages its own metadata using Raft consensus algorithm — built directly into the broker. No external dependency. Simpler, faster failover, supports millions of partitions vs ~200k practical limit in Zookeeper mode.

Kafka Internals — Under the Hood
How replication, leader election, and ISR actually work
💡 Why learn internals: When things go wrong in production — rebalances, lag spikes, broker failures — understanding internals is the difference between fixing it in 10 minutes vs 4 hours.
REPLICATION INTERNALS Topic: orders, 3 partitions, replication-factor=3, 3 brokers Partition 0: Broker 1 → LEADER (accepts reads and writes) Broker 2 → FOLLOWER (replicates from leader) Broker 3 → FOLLOWER (replicates from leader) Partition 1: Broker 2 → LEADER Broker 1 → FOLLOWER Broker 3 → FOLLOWER Partition 2: Broker 3 → LEADER Broker 1 → FOLLOWER Broker 2 → FOLLOWER Leaders are spread across brokers automatically for load balance
How ISR (In-Sync Replicas) works
ISR = set of replicas that are caught up to the leader

Normal state (all replicas healthy):
  ISR = [Broker1(leader), Broker2, Broker3]
  acks=all → wait for all 3 to confirm → safe

Broker 3 falls behind (slow network):
  ISR = [Broker1(leader), Broker2]     ← Broker3 removed from ISR
  acks=all → wait for only 2 to confirm
  Broker3 catches up → rejoins ISR

Broker 2 crashes:
  ISR = [Broker1(leader)]
  min.insync.replicas=2 → PRODUCER THROWS NotEnoughReplicasException
  ← protects against data loss when too few replicas are up

Key config: min.insync.replicas = replication.factor - 1
Production: RF=3, min.ISR=2 → can lose 1 broker, still write
LEADER ELECTION (KRaft mode) Broker 1 (leader for partition 0) crashes │ ▼ KRaft Controller detects heartbeat timeout │ ▼ Controller checks ISR: [Broker2, Broker3] both in-sync │ ▼ Controller elects Broker2 as new leader (first in ISR) │ ▼ Metadata update propagated to all brokers │ ▼ Producers/consumers reconnect to Broker2 Total failover time: ~30 seconds (vs minutes in Zookeeper mode)
How producer writes work internally
Step 1 — Producer asks broker for metadata
Producer → any broker: "who is leader for topic orders partition 0?"
Broker → Producer: "Leader is Broker1 at 192.168.1.1:9092"

Step 2 — Producer batches messages
Producer accumulates messages in RecordAccumulator
Sends batch when: batch.size reached OR linger.ms elapsed

Step 3 — Leader receives batch
Broker1 (leader) appends to partition log on disk
Assigns offset to each message (monotonically increasing)

Step 4 — Replication (acks=all)
Broker2 fetches from leader → appends to replica log
Broker3 fetches from leader → appends to replica log
Both send acknowledgment to leader

Step 5 — Leader responds to producer
Leader sends: "Stored at partition=0 offset=1247"
Producer callback fires: result.getRecordMetadata().offset()
How consumer reads work internally
Step 1 — Consumer joins group
Consumer → GroupCoordinator: JoinGroup request with group-id
GroupCoordinator assigns consumer as member, assigns partitions
(Partition assignment = rebalance)

Step 2 — Consumer fetches messages
Consumer → Leader of partition: FetchRequest(offset=1248, maxBytes=1MB)
Leader → Consumer: Records from offset 1248 to 1300

Step 3 — Consumer processes
Your @KafkaListener method runs
Business logic executes

Step 4 — Consumer commits offset
ack.acknowledge() → OffsetCommitRequest to GroupCoordinator
GroupCoordinator stores: group=fraud-service, partition=0, offset=1301
Next poll starts at 1301

Step 5 — Heartbeat (separate thread)
Consumer sends heartbeat every heartbeat.interval.ms (default 3s)
If no heartbeat for session.timeout.ms (default 10s) → rebalance triggered

// Punctuator runs on a schedule WITHIN the stream processing thread
// Use for: periodic aggregation flush, cleanup, alerts
public class FraudAlertTransformer
    implements Transformer<String, TransactionEvent, KeyValue<String, FraudAlert>> {

  private ProcessorContext context;
  private KeyValueStore<String, Long> txnCountStore;

  @Override
  public void init(ProcessorContext context) {
    this.context = context;
    txnCountStore = context.getStateStore("txn-count");

    // Schedule: check for velocity attacks every 60 seconds
    context.schedule(
      Duration.ofSeconds(60),
      PunctuationType.WALL_CLOCK_TIME,
      (timestamp) -> {
        KeyValueIterator<String, Long> iter = txnCountStore.all();
        while (iter.hasNext()) {
          KeyValue<String, Long> entry = iter.next();
          if (entry.value > 10) {  // >10 txns in 60s = velocity attack
            context.forward(entry.key,
              FraudAlert.velocityAlert(entry.key, entry.value));
          }
        }
        iter.close();
      }
    );
  }

  @Override
  public KeyValue<String, FraudAlert> transform(String key, TransactionEvent txn) {
    Long count = txnCountStore.get(key);
    txnCountStore.put(key, count == null ? 1L : count + 1);
    return null;  // punctuator handles output
  }
}

Kafka Design Patterns
Event notification vs event-carried state vs event sourcing
Pattern 1 — Event NotificationLIGHTWEIGHT
▶
Event contains minimal data — just enough to notify. Consumer calls back to source for full details. Smallest event size, but adds extra network call.
// Event Notification — minimal payload
public record OrderCreatedNotification(
  String orderId,           // just the ID
  Instant occurredAt        // when it happened
  // NO order details — consumer must call order-service REST API
) {}

// Consumer must make a REST call to get full order
@KafkaListener(topics = "order-notifications")
public void handle(OrderCreatedNotification notif, Acknowledgment ack) {
  Order full = orderServiceClient.getOrder(notif.orderId()); // extra call
  processOrder(full);
  ack.acknowledge();
}
⚠️Adds coupling — consumer must know how to call producer. Order Service must be up when consumer processes.
Pattern 2 — Event-Carried State TransferRECOMMENDED
▶
Event contains ALL data consumer needs. No callback required. Consumer is fully autonomous. This is what NeoBank uses — TransactionCompletedEvent carries everything notification and ledger services need.
// Event-Carried State — full payload, consumer self-sufficient
public record OrderCreatedEvent(
  String orderId,
  String customerId,
  String customerEmail,    // notification needs this — include it
  String customerName,     // notification needs this — include it
  BigDecimal amount,
  List<OrderItem> items,   // ledger needs this — include it
  Instant createdAt,
  String correlationId
) {}

// Consumer needs NOTHING else — fully decoupled
@KafkaListener(topics = "order-events")
public void handle(OrderCreatedEvent event, Acknowledgment ack) {
  emailService.sendConfirmation(event.customerEmail(), event.orderId());

}

@Service
@Slf4j
@RequiredArgsConstructor
public class FraudDetectionService {

  private final ChatClient chatClient;

  @CircuitBreaker(name = "ollama-ai", fallbackMethod = "fallbackAnalysis")
  @Retry(name = "ollama-ai")
  public FraudAnalysisResult analyzeTransaction(TransactionInitiatedEvent event) {
    return chatClient.prompt()
      .user(buildPrompt(event))
      .call()
      .entity(FraudAnalysisResult.class);
  }

  // Fallback when circuit is OPEN (AI is down)
  public FraudAnalysisResult fallbackAnalysis(
      TransactionInitiatedEvent event, Exception ex) {

    log.warn("AI circuit open for txn={} — using rule-based fallback",
      event.getTransactionId());

    // Rule-based fallback — no AI, basic rules only
    boolean suspicious = event.getAmount()
      .compareTo(new BigDecimal("50000")) > 0;

    return FraudAnalysisResult.builder()
      .verdict(suspicious ? "SUSPICIOUS" : "LEGITIMATE")
      .confidenceScore(0.5)
      .riskScore(suspicious ? 6 : 2)
      .recommendedAction(suspicious ? "FLAG_FOR_REVIEW" : "APPROVE")
      .reasons(List.of("AI unavailable — rule-based analysis"))
      .build();
  }
}
